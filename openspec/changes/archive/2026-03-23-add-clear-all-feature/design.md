## Context

当前系统已实现规则配置的增删改查、批量删除和Excel导入功能。数据库使用 MySQL，前端为原生 HTML/JS，后端为 Express。

本次变更需要新增"一键清空所有配置"功能，这是一个高风险操作，需要特别注意安全性和用户体验。

## Goals / Non-Goals

**Goals:**
- 提供一键清空所有规则配置的快捷方式
- 确保操作有明确的二次确认机制
- 操作后给出清晰的成功/失败反馈
- 保持与现有UI风格一致

**Non-Goals:**
- 不支持部分清空（按条件筛选后清空）
- 不支持撤销/恢复功能
- 不记录操作日志
- 不添加权限控制（依赖现有登录体系）

## Decisions

### 1. 数据库操作方式：使用 DELETE 而非 TRUNCATE

**决策**: 使用 `DELETE FROM rule_config` 而不是 `TRUNCATE TABLE rule_config`

**理由**:
- DELETE 可以返回受影响的行数，便于给用户反馈
- DELETE 是事务安全的，如果后续需要可以加入事务支持
- DELETE 不会重置自增ID计数器，保持数据连续性（虽然这里不是强需求）
- TRUNCATE 在某些MySQL版本中需要特殊权限

**替代方案**: TRUNCATE 性能更好，但缺乏灵活性

### 2. API 设计：独立端点，语义明确

**决策**: 使用 `DELETE /api/rule-config/clear-all` 作为独立端点

**理由**:
- RESTful 语义清晰：DELETE 表示删除，clear-all 表示范围
- 与批量删除 `/api/rule-config/batch` 区分，避免混淆
- 便于后续添加权限控制或审计日志

**替代方案**: POST /api/rule-config/clear（不够RESTful）

### 3. 前端交互：二次确认 + 危险样式

**决策**: 使用专门的确认弹窗，按钮使用红色危险样式

**理由**:
- 红色按钮警示用户这是危险操作
- 弹窗中显示当前数据条数，让用户了解影响范围
- 需要用户明确点击确认，防止误触

**替代方案**: 输入"CONFIRM"确认（过于繁琐）

### 4. 操作后行为：刷新列表并显示Toast

**决策**: 清空成功后刷新表格并显示成功消息

**理由**:
- 与现有删除操作行为一致
- 用户能立即看到数据已被清空

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 误操作导致数据丢失 | 二次确认弹窗 + 显示当前数据条数 |
| 大数据量删除性能问题 | DELETE 操作在MySQL中对于万级数据性能可接受，如有性能问题后续可改为TRUNCATE |
| 并发操作冲突 | 当前系统为单用户管理后台，并发风险低 |

## Migration Plan

无需迁移，纯功能新增。部署后立即可用。
